## 설계 프로세스

$ARGUMENTS에 대해 아래 7단계를 순서대로 수행한다.
각 단계는 앞 단계에서 발생한 질문에 대한 답으로 시작하며, 문서 전체가 하나의 논리적 스토리로 읽혀야 한다.

---

### 1. Problem Definition — "무엇을 왜 해결하는가"

- 현재 상태 (As-Is)와 문제점을 정량적으로 기술
- 5가지 요소: Who / What / When·Where / Impact(수치) / Urgency
- 해결하지 않으면 어떻게 되는지

→ _"이 문제가 실제로 어떤 상황에서 발생하는가?"_

---

### 2. User Scenarios — "누가 어떤 상황에서 겪는가"

Problem에서 도출된 시나리오를 **기능/비기능** 모두 작성한다.

**기능 시나리오**: 핵심 기능이 반드시 포함
```
Actor → Precondition → Flow (핵심 기능) → Exception → Postcondition
```

**비기능 시나리오**: 프로젝트의 특별하거나 핵심적인 요구사항
- 성능: "P99 latency < Xms" → 어떤 구조가 보장하는지
- 아키텍처: "모듈 교체 시 영향 범위 0" → 인터페이스 분리
- 배포: "무중단 GPU 모델 교체" → rolling/canary 전략
- 확장성/장애복구 등

→ _"이 시나리오들을 해결하려면 어떤 접근이 필요한가?"_

---

### 3. 요구사항 정리 — "시나리오에서 도출된 제약"

- 기능적 요구사항 (시나리오에서 추출)
- 비기능적 요구사항 (성능, 확장성, 가용성, 보안 — 시나리오에서 추출)
- 제약 조건 (기술 스택, 인프라, 시간, 비용)
- 불명확한 점은 질문한다

→ _"이 요구사항을 충족하는 접근법은 무엇이 있는가?"_

---

### 4. 선택지 비교 — "대안과 trade-off"

| 접근법 | 장점 | 단점 | 충족 시나리오 | 미충족 시나리오 |
|--------|------|------|-------------|---------------|
| A | ... | ... | S1, S2 | S3 (성능) |
| B | ... | ... | S1, S3 | S2 (복잡도) |

**Trade-off Matrix** (가중치 포함):
| 기준 (가중치) | Option A | Option B |
|--------------|----------|----------|
| 성능 (30%) | ★★★ | ★★☆ |
| 운영 용이성 (25%) | ★☆☆ | ★★★ |
| 가중 점수 | X.X | X.X |

→ _"선택한 설계를 어떤 구조로 구현하는가?"_

---

### 5. 설계 결정 + Architecture — "구조와 근거"

- 선택한 접근법과 **어떤 시나리오의 어떤 요구사항** 때문인지 역참조
- 의식적으로 포기한 것 (trade-off)
- 되돌리기 어려운 결정 vs 나중에 변경 가능한 결정

**Architecture Diagram** (C4 또는 ASCII):
```
[Context] 외부 시스템과의 관계
  └─ [Container] 주요 구성요소
       └─ [Component] 내부 모듈
```

**데이터 흐름**: 핵심 시나리오 1-2개의 흐름을 Mermaid 시퀀스 다이어그램으로

**디렉토리 구조**:
```
project/
  src/
  tests/
  ...
```

**인터페이스 정의**: 핵심 API/함수 시그니처

→ _"이 구조에서 핵심 병목/핵심 로직은 무엇인가?"_

---

### 6. Key Data Structures / Algorithms — "핵심 메커니즘"

- 핵심 자료구조와 선택 이유 (어떤 시나리오의 요구사항 때문인지)
- 알고리즘 복잡도 분석 (시간/공간)
- 성능 특성과 비기능 시나리오와의 연결

→ _"실제 코드에서는 어떻게 구현하는가?"_

---

### 7. 검증 — "시나리오가 모두 충족되는가"

| 시나리오 | 충족 여부 | 검증 방법 |
|---------|----------|----------|
| S1: ... | Y/N | 테스트 / 벤치마크 / 리뷰 |
| S2: ... | Y/N | ... |

- 엣지케이스는 어떻게 처리되는가?
- 확장 시 어디를 변경해야 하는가?
- 미충족 항목이 있으면 다시 4단계로 돌아간다

---

## 과정 표시
매 단계마다 아래 형식으로 과정을 보여준다:
```
[요구] 초당 1000 요청 처리, p99 latency < 50ms (시나리오 S3에서 도출)
[선택지] A: 단일 프로세스 async / B: 마이크로서비스 / C: 서버리스
[비교] A는 단순하지만 수직확장 한계, B는 유연하지만 복잡도↑, C는 cold start 문제
[판단] 현재 규모에서 A가 적합 — 복잡도 대비 성능 충분
[트레이드오프] 나중에 B로 전환 시 API 경계만 유지하면 마이그레이션 가능
```

## 규칙
- 결정을 내려주지 않고, 판단 근거를 제공한다
- 사용자가 선택할 수 있도록 선택지를 명확히 제시한다
- over-engineering 경고: 지금 필요한 것만 설계한다
- 모든 수치/벤치마크에 출처 링크를 첨부한다
