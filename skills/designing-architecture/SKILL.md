## 설계 프로세스

$ARGUMENTS에 대해:

### 1. 요구사항 정리
- 기능적 요구사항 (무엇을 해야 하는가)
- 비기능적 요구사항 (성능, 확장성, 가용성, 보안)
- 제약 조건 (기술 스택, 인프라, 시간)
- 불명확한 점은 질문한다

### 2. 선택지 비교

| 접근법 | 장점 | 단점 | 적합한 경우 |
|--------|------|------|------------|
| A | ... | ... | ... |
| B | ... | ... | ... |
| C | ... | ... | ... |

### 3. 설계 결정
- 선택한 접근법과 근거
- 의식적으로 포기한 것 (trade-off)
- 나중에 변경할 수 있는 결정 vs 되돌리기 어려운 결정

### 4. 상세 설계

**컴포넌트 다이어그램** (ASCII):
```
[Component A] --> [Component B]
      |
      v
[Component C]
```

**데이터 흐름**: 핵심 시나리오 1-2개의 데이터 흐름

**디렉토리 구조**:
```
project/
  src/
  tests/
  ...
```

**인터페이스 정의**: 핵심 API/함수 시그니처

### 5. 검증
- 요구사항이 모두 충족되는가?
- 엣지케이스는 어떻게 처리되는가?
- 확장 시 어디를 변경해야 하는가?

## 과정 표시
매 단계마다 아래 형식으로 과정을 보여준다:
```
[요구] 초당 1000 요청 처리, p99 latency < 50ms
[선택지] A: 단일 프로세스 async / B: 마이크로서비스 / C: 서버리스
[비교] A는 단순하지만 수직확장 한계, B는 유연하지만 복잡도↑, C는 cold start 문제
[판단] 현재 규모에서 A가 적합 — 복잡도 대비 성능 충분
[트레이드오프] 나중에 B로 전환 시 API 경계만 유지하면 마이그레이션 가능
```
- 요구사항을 정량적으로 표현한다
- 선택지를 나열하고 각각의 장단점을 비교한다
- 왜 이 선택을 했는지 판단 근거를 밝힌다
- 포기한 것(trade-off)을 명시한다

## 규칙
- 결정을 내려주지 않고, 판단 근거를 제공한다
- 사용자가 선택할 수 있도록 선택지를 명확히 제시한다
- over-engineering 경고: 지금 필요한 것만 설계한다
