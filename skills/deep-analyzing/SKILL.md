## 분석 프레임워크

$ARGUMENTS에 대해 아래 구조로 분석한다:

### 1. 분석 범위 설정
- 무엇을 분석하는가? (명확한 범위 정의)
- 어떤 관점에서? (성능, 구조, 정확성, 비용, 확장성 등)
- 분석의 깊이는? (사용자에게 확인)

### 2. 구조 분해 (Decomposition)
- 대상을 핵심 구성 요소로 분해한다
- 각 요소의 역할과 책임을 정리한다
- 요소 간 관계를 다이어그램으로 시각화한다

### 3. 심층 분석
분석 유형에 따라 적절한 방법을 적용한다:

**기술/코드 분석:**
- 실행 흐름 추적 (call graph, data flow)
- 복잡도 분석 (시간, 공간)
- 병목점 및 최적화 가능 지점

**논문/이론 분석:**
- 핵심 기여 (contribution)
- 가정과 한계
- 기존 연구와의 차이

**시스템 분석:**
- 리소스 사용 패턴
- 확장성 특성
- 장애 시나리오

**경제/비즈니스 분석:**
- 핵심 변수와 인과 관계
- 시나리오별 결과
- 민감도 분석

### 4. 발견 사항 (Findings)
- 핵심 인사이트 (우선순위 순)
- 예상과 다른 점
- 추가 조사가 필요한 영역

### 5. 결론 및 시사점
- "그래서 이것이 의미하는 것은 ..."
- 실행 가능한 다음 단계

## 과정 표시
매 단계마다 아래 형식으로 과정을 보여준다:
```
[범위] vLLM scheduler의 preemption 전략 분석
[분해] Scheduler → _schedule() → _preempt() → recompute vs swap 분기
[관찰] _preempt()에서 swap 가능 여부를 먼저 확인 (scheduler.py:342)
[판단] swap 실패 시 recompute fallback → KV cache 전체 재계산 비용 발생
[근거] swap 실패율이 높은 이유: block_manager의 free block 부족
[불확실] GPU↔CPU 전송 대역폭 실측 필요 — 현재 이론값 기준
```
- 분석 범위를 먼저 선언한다
- 코드/시스템을 분해하는 과정을 보여준다
- 판단과 근거를 명확히 쌍으로 제시한다
- 불확실한 부분은 `[불확실]`로 표기한다

## 규칙
- 주장에는 근거를 함께 제시한다 (코드 경로, 수치, 출처)
- 코드 분석 시 파일경로:라인번호를 포함한다
- 불확실한 부분은 명시적으로 표시한다
- 시각화(표, 다이어그램)를 적극 활용한다
