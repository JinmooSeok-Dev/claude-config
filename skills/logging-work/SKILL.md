## 작업 기록 프로세스

현재 세션에서 수행한 작업을 분석하여 프로젝트 루트의 `WORKLOG.md`에 엔트리를 추가한다.

### 1. 기존 WORKLOG.md 확인

- 파일이 있으면 끝에 추가한다 (기존 내용 유지)
- 없으면 새로 생성한다 (아래 헤더 포함):
  ```markdown
  # Work Log
  작업 단위별 진행 기록. `/logging-work`로 생성됨.
  ```

### 2. 엔트리 작성

현재 세션의 대화 내용을 분석하여 아래 형식으로 작성한다:

```markdown
---

## [YYYY-MM-DD] 작업 제목
> 작업 유형: 구현 | 버그 수정 | 분석 | 설계 | 학습 | 조사

### 목표
- 이번 작업 단위에서 달성하려 했던 것

### 수행 내용
1. [완료된 세부 작업]
2. [완료된 세부 작업]

### 발생한 문제와 해결

#### 문제 1: [문제 요약]
- **증상**: 무엇이 잘못되었는가
- **원인 파악 과정**: 어떻게 원인을 찾았는가
  - [시도 1] → [결과]
  - [시도 2] → [결과]
- **근본 원인**: 왜 발생했는가
- **해결**: 어떻게 고쳤는가
- **배경 지식**: 이 문제를 이해하는 데 필요한 개념/원리
  > 예: "PagedAttention에서 block size가 KV cache 단편화에 영향을 준다"

### 의사결정 기록
| 결정 사항 | 선택한 방안 | 대안 | 선택 이유 |
|-----------|------------|------|-----------|
| ... | ... | ... | ... |

### 현재 상태
- **완료**: [완료된 것]
- **미완료**: [남은 것]
- **다음 단계**: [바로 이어서 할 수 있는 액션]

### 배운 것 (선택)
- 이번 작업에서 새로 알게 된 것, 향후 참고할 내용
```

### 3. 작성 규칙

- **문제가 없었으면** "발생한 문제와 해결" 섹션을 생략한다
- **의사결정이 없었으면** "의사결정 기록" 섹션을 생략한다
- **학습/조사 작업이면** "배운 것" 섹션을 중심으로 작성한다
- 모든 섹션을 채울 필요 없다 — 해당하는 것만 작성한다
- 간결하게: 핵심만 남기고, 나중에 읽었을 때 맥락을 복원할 수 있을 정도로

### 4. 사용자 확인

엔트리 초안을 먼저 보여주고, 사용자 확인 후 파일에 기록한다.
추가/수정/삭제할 내용이 있는지 묻는다.
